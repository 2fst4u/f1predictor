## 2024-03-23 - Table Hover States on `<tbody>`
**Learning:** For complex tabular data where a single entry (like a driver's prediction) spans multiple `<tr>` tags (e.g., a main row and an expandable/collapsible details row like the Model Mix/Factors influence panel), applying a hover state directly to the `<tr>` causes visual disjointedness. The hover effect cuts off unexpectedly when the user moves their mouse between the two connected rows. Applying the hover state (`hover:bg-white/5`) directly to the wrapping `<tbody>` element effectively groups both the main and detail rows, creating a single, cohesive visual block that tracks perfectly as the user interacts with any part of that entry.
**Action:** When designing complex tables with expandable details or multi-row entries, always structure them with individual `<tbody>` tags per entry and apply hover styles to the `<tbody>` rather than the `<tr>` to ensure grouped interactive highlighting.

## 2024-05-18 - Semantic Grouping for Custom Button Widgets
**Learning:** Using a `<label>` tag to provide a visual heading for a group of interactive elements (like the "Round" selection buttons in `index.html`) is a critical accessibility anti-pattern. Screen readers expect `<label>` to be directly associated with a specific, valid form `<input>` element. When used as a standalone block element purely for styling above a custom widget, it breaks semantic structure and confuses assistive technologies.
**Action:** Always replace standalone `<label>` tags used above custom widget groups with a semantically neutral `<div>` (with an `id`). Then, wrap the interactive group itself in a `role="group"` container and use `aria-labelledby="[id-of-div]"` to cleanly associate the descriptive text with the widget group without violating HTML semantics.
