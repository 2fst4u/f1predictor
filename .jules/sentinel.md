## 2025-05-03 - [CRITICAL] Prevent SSRF Bypass in URL Validation via String Matching
**Vulnerability:** A Server-Side Request Forgery (SSRF) bypass existed in `f1pred/web.py`'s `test_webhook` function. The validation check used `request.url.startswith("https://discord.com/api/webhooks/")`, which allowed an attacker to supply a malformed URL like `https://discord.com@127.0.0.1:80/api/webhooks/`. The `startswith` check would pass, but the underlying `httpx` request would interpret `discord.com` as the basic auth username and route the request to the internal loopback IP `127.0.0.1:80`, enabling internal network probing.
**Learning:** `startswith()` is inherently insecure for URL validation because URL semantics are complex (e.g., `<scheme>://<user>:<password>@<host>:<port>/<path>`). A string prefix match does not guarantee the host is actually what it appears to be.
**Prevention:** Always use proper URL parsing libraries (like `urllib.parse.urlsplit`) to explicitly extract and strictly validate individual URL components (`scheme`, `netloc`, `path`) against allowlists before performing server-side outbound requests based on user input.

## 2025-05-04 - [CRITICAL] Prevent SSRF Bypass in URL Validation via String Matching
**Vulnerability:** A Server-Side Request Forgery (SSRF) bypass existed in `f1pred/web.py`'s `test_webhook` and `f1pred/prediction_manager.py`'s `_send_discord_webhook` function. The validation check used `parsed.netloc not in valid_hosts`, which allowed an attacker to supply a malformed URL like `https://discord.com@127.0.0.1:80/api/webhooks/`. The `netloc` check would pass because `discord.com@127.0.0.1:80` isn't in `valid_hosts` but it routes the request to the internal loopback IP `127.0.0.1:80`, enabling internal network probing.
**Learning:** `netloc` includes basic auth strings (like username and password) and port numbers (like `:80` or `:443`). It is not the same as the isolated domain name or IP.
**Prevention:** Always use `parsed.hostname` when validating domains after using `urllib.parse.urlsplit` to ensure only the actual domain name is compared against allowlists.
## 2025-02-20 - Information Leakage in Webhook Test
**Vulnerability:** An Information Exposure vulnerability existed in `f1pred/web.py`'s `test_webhook` function. The exception handler was returning `str(e)` directly in an `HTTPException(status_code=500, detail=str(e))`.
**Learning:** This could expose internal server state, database connection strings, or third-party service errors to end users, aiding an attacker in further reconnaissance or exploitation.
**Prevention:** To prevent Information Exposure, do not leak raw exception details directly in HTTP responses. Always return generic error messages for 500 status codes while logging the actual exception (`logger.exception()`) for internal debugging purposes.
