## 2024-03-23 - Table Hover States on `<tbody>`
**Learning:** For complex tabular data where a single entry (like a driver's prediction) spans multiple `<tr>` tags (e.g., a main row and an expandable/collapsible details row like the Model Mix/Factors influence panel), applying a hover state directly to the `<tr>` causes visual disjointedness. The hover effect cuts off unexpectedly when the user moves their mouse between the two connected rows. Applying the hover state (`hover:bg-white/5`) directly to the wrapping `<tbody>` element effectively groups both the main and detail rows, creating a single, cohesive visual block that tracks perfectly as the user interacts with any part of that entry.
**Action:** When designing complex tables with expandable details or multi-row entries, always structure them with individual `<tbody>` tags per entry and apply hover styles to the `<tbody>` rather than the `<tr>` to ensure grouped interactive highlighting.## 2024-05-18 - Keyboard accessibility for scrollable wrappers
**Learning:** For users who navigate via keyboard or screen reader, scrollable elements (`overflow-x-auto` or `overflow-y-auto`) must be explicitly made focusable so they can be scrolled without relying on a mouse.
**Action:** Always add `tabindex="0"`, `role="region"`, `aria-label` (or `aria-labelledby`), and visible focus indicators (e.g. `focus:outline-none focus-visible:ring-2`) to the scrollable wrapper element.
